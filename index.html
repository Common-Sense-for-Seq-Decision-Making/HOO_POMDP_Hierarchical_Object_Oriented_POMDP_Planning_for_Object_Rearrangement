Hierarchical Object-Oriented POMDP Planning for Object Rearrangement

Rajesh Mangananvar             Alan Fern             Prasad Tadepalli            

Solves multi-room rearrangement with 10-20 objects and severe partial observability (10-30% initial visibility)

Paper arXiv Code

Abstract

[cite_start]We present an online planning approach and a new benchmark dataset for solving multi-object rearrangement problems in partially observable, multi-room environments[cite: 12]. [cite_start]Current object rearrangement solutions, primarily based on Reinforcement Learning or hand-coded planning methods, often lack adaptability to diverse challenges[cite: 13]. [cite_start]To address this limitation, we propose a Hierarchical Object-Oriented Partially Observed Markov Decision Process (HOO-POMDP) planner that leverages object-factored belief representations for efficient multi-object rearrangement[cite: 14]. [cite_start]This approach comprises of (a) an object-oriented POMDP planner generating sub-goals, (b) a set of low-level policies for sub-goal achievement, and (c) an abstraction system converting the continuous low-level world into a representation suitable for abstract planning[cite: 15]. [cite_start]To enable rigorous evaluation, we introduce MultiRoomR, a comprehensive benchmark featuring diverse multi-room environments with varying degrees of partial observability (10-30% initial visibility), blocked paths, obstructed goals, and multiple objects (10-20) distributed across 2-4 rooms[cite: 16]. [cite_start]Experiments demonstrate that our system effectively handles these complex scenarios while maintaining robust performance even with imperfect perception[cite: 17].

Why POMDPs for Rearrangement? [Expand All]

[cite_start]
▶Embodied AI rearrangement is a fundamental challenge: It's a core task for home assistant robots[cite: 19, 21].
[cite_start]

Multi-object rearrangement with egocentric vision requires integrating perception, planning, navigation, and manipulation, especially in partially observable, multi-room settings common in everyday life[cite: 19, 20, 21].

[cite_start]
▶Real-world tasks are complex: Scenarios involve blocked paths, obstructed goals, and spatial dependencies[cite: 24, 27].

Simple greedy planners or fixed skill sequences can't determine the optimal order to move objects. [cite_start]They fail when spatial reasoning is required, such as moving one object to unblock another's path or goal[cite: 24, 27, 45].

[cite_start]
▶Most agents fail with severe partial observability: Real-world objects are not all visible at the start[cite: 29, 30].
[cite_start]

Existing work often assumes single rooms or high initial visibility (e.g., >60%)[cite: 29, 284]. [cite_start]Realistic tasks, like tidying a house, require agents to actively search for objects that are not initially visible[cite: 30, 31].

The Problem with Existing Approaches [Expand All]

[cite_start]
▶End-to-end RL methods struggle to scale: Pure RL fails in complex, long-horizon rearrangement tasks[cite: 23].
[cite_start]

Reinforcement Learning (RL) methods are not well-suited for the long-horizon, sparse-reward nature of multi-object rearrangement and struggle to scale to complex, multi-room scenarios[cite: 23].

[cite_start]
▶Greedy/Modular planners are not optimal: Hand-coded systems lack the flexibility to handle novel problems[cite: 24].

These approaches often use predetermined skill sequences or simple greedy planners. [cite_start]This restricts their ability to find optimal object interaction orders or handle new challenges like blocked paths without extensive retraining[cite: 24].

[cite_start]
▶Existing benchmarks are too simple: Datasets like RoomR don't test for complex, multi-room challenges[cite: 35].
[cite_start]

Current datasets are often limited to single-room settings with only a few objects[cite: 29, 400]. [cite_start]They fail to test for challenges like severe partial observability, large object counts (10-20), or complex spatial dependencies (blocked paths)[cite: 35, 401].

HOO-POMDP's Key Insight [Expand All]

[cite_start]
▶Hierarchical Planning: Separate high-level strategy (what to do) from low-level execution (how to do it)[cite: 36, 40].
[cite_start]

A high-level POMDP planner handles strategic, abstract decisions (e.g., 'search for object X', 'move object Y to goal')[cite: 37, 38]. [cite_start]This frees it from low-level details, which are handled by specialized policies (e.g., 'navigate to (x,y)', 'pick object')[cite: 39].

[cite_start]
▶Object-Oriented Beliefs: Factor the problem by object to make it scalable[cite: 14, 37].

Instead of one massive, intractable belief state, the system maintains a separate, independent belief (a probability distribution over locations) for each object. [cite_start]This leverages Object-Oriented POMDPs (OO-POMDPs) to make planning scalable[cite: 99, 100, 105].

[cite_start]
▶Jointly Optimize Exploration and Manipulation: The planner reasons under uncertainty to make smart decisions[cite: 38].
[cite_start]

The high-level planner can optimally decide whether it's more efficient to continue searching for a missing object or to rearrange an object it has already found, based on its current belief state[cite: 38].

[cite_start]

Example of complex spatial reasoning: Object 2 blocks the path for Object 3, and Object 3 blocks the goal for Object 1. The planner must find the sequence: Move 2, then 3, then 1[cite: 45].

Method Overview [Expand All]

[cite_start]
▶A cyclic system: Perception → Belief Update → Abstract Planning → Policy Execution[cite: 143].

The agent executes a low-level action, gets a new RGB/Depth observation, and uses it to update its object-factored belief state. [cite_start]This belief is then abstracted into a discrete state for the high-level POMDP planner, which chooses the next sub-goal (abstract action) [cite: 143-148].

[cite_start]
▶Abstraction bridges the gap: Converts continuous belief probabilities into a discrete planner state[cite: 15, 146].

The belief state (probabilities over all grid cells) is abstracted into a discrete state for the planner. [cite_start]This includes sampling reachable `pick` locations (based on belief) and `place` locations (near goals/receptacles)[cite: 146, 208, 209, 218].

[cite_start]
▶Low-Level Policies execute sub-goals: A* for navigation, trained RL agents for interaction[cite: 181, 186].
[cite_start]

Abstract actions like `Move_AB` or `Rotate` are executed by an A* pathfinder[cite: 185, 189]. [cite_start]The `PickPlace` action is a 3-stage policy: an RL model to Pick, A* to navigate, and another RL model to Place[cite: 186, 193].

Results [Expand All]

[cite_start]
▶Outperforms all baselines: HOO-POMDP is more robust and flexible than RL (VRR) and hand-coded (FHC, MSS) methods[cite: 276].

Our principled planning approach significantly outperforms baselines. [cite_start]VRR (pure RL) fails to scale[cite: 277]. [cite_start]FHC (heuristic) fails to handle detector uncertainty[cite: 278]. [cite_start]MSS (rigid explore-then-rearrange) cannot handle blocked paths[cite: 279].

[cite_start]
▶Introduces MultiRoomR: A new, challenging benchmark with 2-4 rooms, 10-20 objects, and blocked paths[cite: 237].
[cite_start]

We introduce the MultiRoomR dataset (400 configurations) to test scenarios existing datasets don't: severe partial observability (10-30% initial visibility), larger object counts, and complex spatial dependencies (50% of scenes have blocked paths) [cite: 237, 403-406].

[cite_start]
▶Handles uncertainty robustly: Performance is close to the "Perfect Detector" oracle, even with 50-60% detector success[cite: 270, 273].

Despite imperfect perception, HOOP solves many problems. [cite_start]The small performance gap between HOOP and the Perfect Detector (PD) oracle shows that our POMDP planner effectively handles partial observability and detector failures [cite: 270-273].

Key Findings from Ablations [Expand All]

[cite_start]
▶Hierarchical planning is critical: A "flat" planner (HOOP-HP) fails completely, with 0-9% success[cite: 275, 245].
[cite_start]

The `HOOP-HP` ablation, which removes the hierarchy and plans directly in the low-level action space, shows a 'substantial performance gap'[cite: 275]. [cite_start]Table 1 shows its Scene Success drops to near-zero on complex problems, proving the low-level action space is intractable[cite: 245].

[cite_start]
▶Strategic lookahead is essential: A greedy MCTS planner (depth 1) fails to solve almost any problems[cite: 476, 477].
[cite_start]

As shown in Table 3, an MCTS search depth of 1 (HOOP-MCTS 1) is too myopic and fails to get a useful reward signal[cite: 477, 488]. [cite_start]Our full model uses a depth of 12, showing that lookahead is necessary to make strategic decisions[cite: 476].

[cite_start]
▶Principled planning > simple heuristics: The FHC baseline is brittle and cannot handle uncertainty or complex paths[cite: 278, 279].
[cite_start]

The `FHC` (Frontier Exploration) baseline uses a simple 'explore-then-rearrange' heuristic[cite: 249, 251]. [cite_start]It fails to handle object detection failures because it's not belief-based, and it cannot reason about complex spatial dependencies like blocked paths[cite: 278, 279].

Technical Details [Expand All]

[cite_start]
▶Perception: YoloV10 trained on 10k images from 500 AI2Thor scenes[cite: 374, 376].
[cite_start]

We trained a YoloV10-Large (25M parameters) for 500 epochs[cite: 376, 377]. [cite_start]The resulting True Positive (TP) and False Positive (FP) rates for each object class (see Table 2 in paper) are used to build the probabilistic observation model for the POMDP[cite: 383, 410].

[cite_start]
▶Planner: PO-UCT (Partially Observable UCT) searches over action-observation histories[cite: 167, 168].

The planner samples a state from the current belief, builds a search tree using UCT action selection, and runs rollouts to estimate value. [cite_start]The full algorithm is provided in the paper's appendix (Algorithm 3) [cite: 168-172, 483].

[cite_start]
▶Low-Level Policies: A* for navigation, PPO-trained RL agents for Pick/Place[cite: 185, 186, 193].
[cite_start]

The RL policies were trained for 5 million steps using PPO[cite: 491, 492]. [cite_start]The Pick and Place policies are given navigation actions so they can finely adjust the agent's position before an interaction, making them more robust[cite: 504].

[cite_start]
▶New Dataset (MultiRoomR): 400 configurations, 2-4 rooms, 10-20 objects[cite: 237, 238, 438].

We introduce this new benchmark to test more realistic scenarios. Key features include:

    [cite_start]
  • Severe Partial Observability: Only 10-30% of objects are visible at the start[cite: 237, 443].
    • [cite_start]
  • Complex Dependencies: 50% of scenes contain blocked paths, and others include blocked goals and object swaps[cite: 405, 469, 470].
    • [cite_start]
  • Scalability: Tasks involve 10, 15, or 20 objects, forcing planners to be efficient[cite: 237, 404].

Why This Matters [Expand All]

[cite_start]
▶A scalable, robust solution for a core Embodied AI task: Planning for home assistant robots[cite: 19, 21].

This provides a practical and scalable method for multi-object rearrangement, a key skill for any future home assistant robot. [cite_start]It robustly handles uncertainty and complexity where pure RL and simple heuristics fail[cite: 22, 276, 280].

[cite_start]
▶Integrates hierarchical abstraction with formal POMDP planning: A principled way to combine strategy and control[cite: 36, 40].

Our work shows how to effectively combine high-level, strategic planning under uncertainty (POMDP) with low-level continuous control (RL/A* policies). [cite_start]The hierarchy and abstraction are key to making the problem tractable[cite: 39, 40, 275].

[cite_start]
▶A new, challenging benchmark (MultiRoomR) for the community: Pushes research beyond simple problems[cite: 16, 35].

By releasing MultiRoomR, we move the community beyond simplistic single-room datasets. [cite_start]We provide a benchmark to test systems on more realistic challenges, including severe partial observability and complex spatial dependencies [cite: 35, 403-407].
